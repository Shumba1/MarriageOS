# Mission

## Project
MarriageOS

## Phase
Phase 0 — Source Conditioning and Canonical Lock

## Mission
Build the governed source-conditioning layer for MarriageOS so downstream content generation and later implementation are grounded in canonical doctrine, structured extracts, contradiction-aware synthesis, and output-ready dossiers.

## Desired outcome
Turn the currently available Marriage Project materials into a clean, traceable, contradiction-aware source base that can reliably support later content generation, implementation planning, and app build work without rereading the entire archive or guessing doctrine.

## What Phase 0 is
Phase 0 is a source-pipeline phase.

It exists to:
- identify canonical authority
- normalize currently available source materials
- generate atomic extract cards
- deduplicate overlaps
- flag contradictions and drift
- assemble target dossiers
- verify dossier readiness for downstream work

## What Phase 0 is not
Phase 0 is not:
- app build
- UI build
- route implementation
- final polished content generation at scale
- feature production
- browser automation work

## Canonical authority
The canonical governing layer for this project is:

1. `docs/MASTER_SPEC.md`
2. `docs/DRIFT_REGISTER.md`
3. `src/config/theme.ts`
4. `src/config/page-data.ts`

If any raw source, extract, dossier, or feeder file conflicts with this layer, the canonical layer wins.

## Active feeder sources
These are valid support sources, but they do not overrule canonical authority:

- `Marriage Project.md`
- `Marriage Project Revised premium UX spec.md`
- `Marriage Project Addendum - splitting the knowledge layer into two different component responsibilities.md`
- `Marriage Project Version One Code Samples.md`
- `Marriage Project Builder-ready Token file and page-by-page component spec.md`

## Reference-only source
- `Marriage Project Version One PWA.md`

This may be used for historical context only, not as implementation truth.

## Strong constraints
- Repair remains the centre of V1.
- Knowledge is supporting infrastructure, not the centre of the product.
- Perspectives are parallel interpretation layers, not adversarial camps.
- Do not go from raw source directly to final output.
- Every meaningful output must be written back into the repo at a defined path and in a defined format.
- Every contradiction must be logged explicitly.
- No source file is treated as equal in authority until classified.

## Required pipeline
Use this sequence only:

1. Raw source material
2. Structured extracts
3. Deduplicated extract set
4. Target dossier
5. Verification pass

## Exit criteria for Phase 0
Phase 0 is complete only when:

- canonical authority is explicit and locked
- all currently available source materials are classified and placed correctly
- normalized source records exist for each major input
- atomic extract cards exist for the current materials
- duplicate extracts are merged or flagged
- contradictions are visible in one place
- at least one dossier can be built for each priority target object type
- dossier readiness can be judged without rereading the full archive

## Priority target objects
The first dossier-ready targets should be drawn from the highest-value clusters already suggested by the project materials:

- drift
- resentment
- pressure loop / intimacy mismatch
- one-sided initiation
- postpartum / touched out
- mental load / manager mode
- desire maintenance
- external pull / affair risk

## Definition of done
MarriageOS Phase 0 is done when the repo contains a trustworthy, governed source base that a builder or model can use for downstream work without confusing historical prose, feeder notes, and canonical truth.
