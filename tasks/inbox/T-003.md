# T-003 — Generate atomic extract cards from the current source set

## Objective
Turn the currently available MarriageOS materials into atomic extract cards that preserve doctrine, constraints, architecture, UX, and product logic without burying them in long prose files.

## Why it matters
Downstream dossier work depends on extract-level reasoning, not rereading whole documents.

## Inputs
- `docs/source-records/*`
- all current major source files
- `docs/pipeline/prompts/01-source-normaliser.md`
- `docs/pipeline/templates/extract-card-template.yaml`
- `docs/pipeline/schemas/extract.schema.json`

## Worker role
Driver

## Output path
`sources/derived/extracts/`

## Output format
One extract card per atomic insight, saved as `.yaml` or `.md`, consistent with the extract schema.

## Acceptance criteria
- extracts are atomic, not essay-like
- every extract preserves source attribution
- extracts are grouped by meaningful themes such as doctrine, UX, architecture, constraints, and pipeline logic
- extract cards are suitable for later dedupe
- extracts do not smuggle in unsupported inferences
- extract wording does not distort source intent

## Observability hooks
- extract_count
- source_to_extract_ratio
- unattributed_extract_count
- doctrine_extract_count
- ux_extract_count
- architecture_extract_count
- extract_quality_ready = true/false

## Status
inbox

## Notes
Do not deduplicate in this task. Capture first, dedupe later.
