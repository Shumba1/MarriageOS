# T-005 — Build the first target dossiers

## Objective
Assemble dossier-ready inputs for one output object at a time from the canonical layer plus deduplicated extracts.

## Why it matters
Dossiers are the bridge between source conditioning and downstream content generation or implementation.

## Inputs
- `docs/MASTER_SPEC.md`
- `docs/DRIFT_REGISTER.md`
- `docs/README_ACTIVE_SET.md`
- `sources/derived/extracts-deduped/`
- `docs/contradictions.md`
- `docs/pipeline/prompts/03-dossier-builder.md`
- `docs/pipeline/templates/dossier-template.md`
- `docs/pipeline/schemas/dossier.schema.json`
- `docs/pipeline/examples/pressure-loop-dossier-example.md`

## Worker role
Driver

## Output path
`sources/derived/dossiers/`

## Output format
One markdown dossier per target object.

## Initial target objects
Build dossiers for:
- one repair module
- one perspective guide
- one knowledge article
- one product/tool page

## Acceptance criteria
- each dossier is for one object only
- each dossier is canonical-first
- each dossier preserves source traceability
- unresolved contradictions are visible inside the dossier
- dossiers are specific enough for downstream generation or implementation planning
- dossiers do not collapse multiple object types into one generic brief

## Observability hooks
- dossier_count
- dossier_by_type
- traceability_complete_count
- dossier_with_unresolved_conflict_count
- dossier_build_complete = true/false

## Status
inbox

## Notes
Do not try to build every dossier in one pass. Build the first four target objects only.
